Understanding JSON Web Tokens (JWT) with a Simple Example

Security Notes

Sep 21, 2025

When I first looked into JSON Web Tokens (JWT), my reaction was: this is just a string with dots — how can it secure anything? The trick is how that string is constructed: a header, a payload, and a signature, Base64URL-encoded and joined with periods.

The header declares the algorithm and type, e.g. {"alg":"HS256","typ":"JWT"}. The payload carries claims (like who you are and your role), e.g. {"user":"alice","role":"admin"}. Both are encoded, then the server creates a signature with its secret key to prevent tampering.

Step 1 — Defining the parts

{
  "header":  {"alg":"HS256","typ":"JWT"},
  "payload": {"user":"alice","role":"admin"}
  }

Step 2 — Encoding the parts

base64url(header)  = eyJhbGciOiAiSFMyNTYiLCAidHlwIjogIkpXVCJ9
base64url(payload) = eyJ1c2VyIjogImFsaWNlIiwgInJvbGUiOiAiYWRtaW4ifQ

Step 3 — Creating the signature

signature = HMACSHA256( header + "." + payload, <server-secret> )

Step 4 — The final JWT

eyJhbGciOiAiSFMyNTYiLCAidHlwIjogIkpXVCJ9
eyJ1c2VyIjogImFsaWNlIiwgInJvbGUiOiAiYWRtaW4ifQ
p4l2jMZr7vP5Zp5yGk6czhZW3fWfGgvAxQs0LwC9ZtU

The important part is the signature: it’s created using a secret key known only to the server. Anyone can decode the first two parts and read the claims, but without the secret they cannot generate a valid signature. That’s why JWTs rely on strong, random secrets — brute forcing a long, random key is simply impractical.

To make it less abstract, think of JWTs like an airline boarding pass. The header is the format, the payload says who you are and where you’re headed, and the signature is the official stamp from the airline. Anyone can read your name and flight info, but only the airline can issue the stamp. Without it, you’re not getting past the gate.
